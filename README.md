# Fact-Checking Agent with a Fine-Tuned Grounding Judge

A claim-verification system built around a **fine-tuned, locally-served LLM judge**. The judge is a Qwen3 8B model post-trained with an **SFT → DPO** pipeline to do one thing well: decide a verdict *only* when it can be grounded in retrieved evidence, and **refuse otherwise**. The model is deployed as the reasoning core of a LangGraph agent that decomposes a claim into sub-claims, retrieves evidence for each in parallel, judges them, and aggregates a final verdict.

> The interesting part of this project is not the agent — it's the judge. Off-the-shelf Qwen3 8B answers everything, even when the evidence doesn't support an answer. After fine-tuning it learns to refuse appropriately, lifting **pass rate from 0.47 → 0.73** and **refusal precision/recall from 0 → 0.85/0.85** while keeping over-refusal at **3.75%**.

---

## Highlights

- **Post-trained a grounding judge (SFT → DPO, LoRA, TRL)** on Qwen3 8B to ground answers in evidence and refuse when context is insufficient.
- **Quantified the improvement with Ragas + custom refusal heuristics** against the untuned base model — not vibes, numbers.
- **Shipped it locally.** Quantized to `Q5_K_M` GGUF with llama.cpp so it runs on a 16 GB MacBook Air, served via Ollama. Cost-free and private at inference time.
- **Deployed inside an agentic system.** A LangGraph fan-out/fan-in graph applies the judge per sub-claim with parallel evidence retrieval.

---

## The Problem

A useful fact-checker has to know when *not* to answer. A model that confidently returns a verdict for every claim — including ones the evidence can't support — is worse than useless, because its confident-but-ungrounded answers are indistinguishable from its correct ones. The base Qwen3 8B exhibits exactly this failure: in evaluation it **never refused** (refusal precision and recall of 0), answering every query regardless of whether the retrieved context supported it.

The goal of this project was to fix that behavior at the model level — teach the judge to ground its verdicts in evidence and refuse when grounding isn't possible — and then build an agent that puts the improved judge to work.

## Post-Training Pipeline

**Base model:** Qwen3 8B (BF16).

**Stage 1 — Supervised Fine-Tuning (SFT)**
- **Data:** ~2K examples drawn from a combination of **MS MARCO** and **SQuAD v2**.
- **Task format:** `query + context → response`.
- **Refusal coverage:** ~20% of examples were refusals where the context did not contain enough information to answer the query — teaching the model that "I can't answer this from the given context" is a valid, desired output.

**Stage 2 — Direct Preference Optimization (DPO)**
- **Preference pairs:** *rejected* responses were generated by the untuned **Qwen3 8B base model**; *chosen* responses reflect the grounded/refusal behavior.
- **Objective:** push the model toward answering only when the response can be grounded in the provided context, and toward correctly refusing when it cannot.

**Training setup:** LoRA adapters for both the SFT and DPO stages, implemented with **TRL**.

**Deployment:** the resulting model was merged and **quantized to `Q5_K_M` GGUF using llama.cpp**, so it fits in 16 GB of memory and runs locally through **Ollama** (`qwen3-dpo-q5-gguf`). All checkpoints (base-SFT, DPO, GGUF) are published on the Hugging Face Hub.

## Evaluation

The fine-tuned model was evaluated against the **untuned Qwen3 8B baseline**. Quality was measured with **Faithfulness** and **Answer Accuracy** from [Ragas](https://docs.ragas.io/), aggregated into a combined **pass rate**. Refusal behavior was measured separately with heuristics: **refusal precision/recall** (does it refuse when it should?) and **over-refusal rate** (does it refuse when it shouldn't?).

| Metric              | Baseline | SFT    | DPO    |
|---------------------|:--------:|:------:|:------:|
| Pass rate           | 0.47     | 0.73   | 0.73   |
| Refusal precision   | 0.00     | 0.84   | 0.85   |
| Refusal recall      | 0.00     | 0.80   | 0.85   |
| Over-refusal rate   | 0.00     | 0.0375 | 0.0375 |

**How to read this:**
- **SFT drove the answer-quality gain** — pass rate jumped from 0.47 to 0.73 (a ~55% relative improvement) and the model went from never refusing to refusing with high precision.
- **DPO sharpened refusal calibration** — refusal recall improved from 0.80 to 0.85 (catching more of the cases it *should* refuse) with no regression in pass rate or over-refusal.
- **Over-refusal stayed low (3.75%)**, so the gains in refusal behavior didn't come at the cost of the model becoming uselessly cautious — the trade-off most refusal-tuning gets wrong.

> Note: the baseline's `0.00` refusal scores reflect that it effectively never refuses, so there are no true positives to score precision/recall against.

## Why Fine-Tune a Local Judge Instead of Prompting a Frontier Model?

Three reasons:

1. **Cost** — the judge runs on every sub-claim of every request. A local model makes that free at inference time instead of paying per token.
2. **Privacy** — judging runs entirely on-device through Ollama; no claim or evidence leaves the machine.
3. **Learning exercise** — the project was a deliberate end-to-end run through a modern post-training stack (SFT → DPO → quantization → deployment → evaluation).

The decomposition step still uses a hosted model (`gpt-4o-mini`) where its broad capability helps and the call count is low, while the high-frequency judging is handled locally — a pragmatic split of work across hosted and local models.

## Agent Architecture

The agent is a [LangGraph](https://langchain-ai.github.io/langgraph/) graph that applies the fine-tuned judge across a claim's sub-claims in parallel.

```
                ┌──────────────┐
   raw claim →  │ ingest_claim │  normalize input
                └──────┬───────┘
                       ▼
              ┌─────────────────┐
              │ decompose_claim │  gpt-4o-mini → structured sub-claims
              └────────┬────────┘
                       ▼
            continue_to_retrieve   ── Send() fan-out: one sub-graph per sub-claim
                       ▼
        ┌──────────────────────────────┐
        │            SUB-GRAPH          │   (runs in parallel per sub-claim)
        │  ┌────────────────────────┐   │
        │  │   retrieve_evidence    │   │   Tavily web search
        │  └───────────┬────────────┘   │
        │              ▼                 │
        │  ┌────────────────────────┐   │
        │  │      judge_claim       │   │   fine-tuned Qwen3 judge → verdict,
        │  └────────────────────────┘   │   confidence, grounding sentence
        └──────────────┬───────────────┘
                       ▼
              check_confidence          ── low-confidence sub-claims retried
                       ▼
             ┌────────────────────┐
             │ aggregate_verdicts │  refuted > insufficient > supported
             └─────────┬──────────┘
                       ▼
              ┌─────────────────┐
              │  format_output  │
              └─────────────────┘
```

**Key design points:**
- **Fan-out/fan-in via the `Send` API** — each sub-claim gets its own sub-graph instance, so retrieval and judging run concurrently.
- **Structured outputs** — both the decomposer (`SubClaim`) and judge (`ClaimCheck`) use Pydantic schemas via `with_structured_output`, so node outputs are typed and validated.
- **Custom state reducers** — evidence and judge results accumulate across parallel branches (`operator.add`), while `retry_count` uses a custom overwrite reducer.
- **Conservative aggregation** — a single `refuted` sub-claim refutes the overall claim; `insufficient evidence` takes precedence over `supported`.

## Setup & Usage

### Prerequisites
- Python 3.10+
- [Ollama](https://ollama.com/) running locally with the fine-tuned judge model available as `qwen3-dpo-q5-gguf`
- API keys for OpenAI (decomposer) and Tavily (web search)

### Install
```bash
git clone <repo-url>
cd factcheck-agent
python -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt   # see note below
```

### Environment
Create a `.env` file with:
```
OPENAI_API_KEY=your_key_here
TAVILY_API_KEY=your_key_here
```

### Run as an API
```bash
uvicorn app:app --reload
```
Then open `http://127.0.0.1:8000/docs` and POST a claim to `/invoke`.

## Limitations & Future Work

- **Retrieval depth** — currently uses a single search result per sub-claim; multi-source retrieval and source-credibility weighting would improve robustness.
- **No source ranking** — all evidence is treated as equally trustworthy.
- **Temporal claims** — claims like "currently has the highest market share" depend on retrieval freshness and aren't explicitly handled.
- **Agent-level evaluation** — the *judge* is rigorously evaluated, but end-to-end agent accuracy on a labeled claim set is future work.
- **Error handling** — external calls (search, model inference) assume the happy path and need graceful degradation.
